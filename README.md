# micro-SFA

**Phase（事実チェック）× BANTスコア（客観指標）** で案件を管理する超軽量SFA。

営業の「主観による確度ランク」問題を解決し、誰が見ても同じ基準でヨミ（見込み）を判断できるようにする。

|---------|-------------|

## 特徴

- **Phase × BANT で自動ヨミ判定** — 営業の自己申告なしで A/B/C を算出
- **外部依存ゼロ** — Go `net/http` + Vanilla JS ES Modules + Canvas API
- **1コマンドで起動** — `go run main.go`
- **localStorage** — バックエンドDB不要、デモ即開始
- **AIで改造しやすい** — 関心分離・定数集約・純粋関数

## データモデル

### Phase（事実チェック・主観禁止）

| Phase | 条件（事実） |
|-------|------------|
| 1 | 議事録共有・課題合意済み |
| 2 | 見積書送付・受領確認済み |
| 3 | 決裁権者出席の会議日時確定 |
| 4 | 発注意思受領・契約書確認中 |

### BANTスコア（0〜8点）

| 項目 | 0点 | 1点 | 2点 |
|------|-----|-----|-----|
| Budget | 未確認 | 概算予算枠を確認 | 確定予算・金額ラインを把握 |
| Authority | 窓口担当者のみ | 起案者・影響者を把握 | 最終決裁権者を特定または同席 |
| Needs | ふんわりした興味 | 明確なペインをヒアリング済 | RFP等として要件定義済 |
| Timeframe | 時期未定 | 大枠の時期が判明 | デッドライン（日付）が確定 |

### ヨミ区分（自動計算）

| ヨミ | 条件 | 計上率 |
|------|------|--------|
| A（確実） | Phase≧3 かつ BANT≧6 | 90% |
| B（勝負） | Phase≧2 かつ BANT≧4 | 50% |
| C（観察） | それ以外 | 10% |

## カンバンの使い方

カンバンは「チーム全体の案件が今どのフェーズにいるか」を一覧するビューです。営業担当者は主にマイページで作業しますが、マネージャーがチームの状況を把握・指示するときに使います。

### 画面構成

```
[ 未Phase ] [ Phase 1 ] [ Phase 2 ] [ Phase 3 ] [ Phase 4 ] [ 受注済み ]
```

各カラムには「そのフェーズに現在いるアクティブ案件」だけが表示されます。失注済みは非表示（マイページの失注タブで確認）。

### カードの読み方

```
┌─────────────────────────────┐
│ ⚠ BANT不足          ← 警告バッジ（Phase に対して BANT が低すぎる場合）
│ 株式会社〇〇 導入提案       ← 案件名
│ 田中 一郎   ¥1,200,000     ← 担当者・売上金額
│ 🏢 社内検討中              ← ボール位置（今誰が動く番か）
│ ████████░░  BANT 6/8       ← BANTスコアバー（赤/黄/緑）
│ 編集 →                     ← クリックで案件編集へ
└─────────────────────────────┘
```

**BANTバーの色の意味**

| 色 | スコア | 意味 |
|----|--------|------|
| 🟢 緑 | 6〜8点 | ヨミA圏。情報収集ほぼ完了 |
| 🟡 黄 | 4〜5点 | ヨミB圏。引き続き情報収集が必要 |
| 🔴 赤 | 0〜3点 | ヨミC圏。まだ初期段階 |

**⚠ BANT不足バッジ**が出ている案件は、フェーズが進んでいるわりにBANTスコアが低い状態です。「Phase は高いのに顧客情報が薄い」という危険サインなので、優先的に確認してください。

**ボール位置バッジ**は「今この案件で次に動くのは誰か」を示します。🏢 社内 = 自社が動く番、👤 顧客 = 顧客の返事待ち、⚙ 社内調整 = 社内で承認待ち。

### フィルター（マネージャー・admin のみ）

画面右上の2つのセレクトで絞り込めます。

- **部署フィルター** — 部署単位でパイプラインを確認
- **担当者フィルター** — 個人の案件だけを表示。1on1 や週次レビューで使う

### よくある使い方

1. **週次レビュー** — フィルターで担当者を絞り、「Phase 2 に長くいる案件」を拾い上げてコーチングのネタにする
2. **ヨミ会前の確認** — ⚠バッジが多い担当者に事前にBANT更新を促す
3. **受注確認** — 「受注済み」カラムに案件が増えていることをチームで視覚的に共有する
4. **案件作成** — 右上「＋ 案件作成」から直接新規作成。保存後カンバンに戻る

## 設計判断：想定受注日を必須にする理由

「まだ日程が見えない案件には入力できない」という声は必ず出る。しかしこのSFAでは、想定受注日を**任意にしない**。

### 想定受注日は「予測」ではなく「コミットメント」である

SFAに日付を入力する行為は、単なる見込み日の記録ではない。担当者が「この案件はこの時期に受注できると現時点で判断した」という**意思表明**であり、そこから行動計画が逆算される。

```
想定受注日
  ↓ 逆算
今月中にやること / 今週やること / 今日やること
  ↓ 調整
他の案件のネクストアクションとの競合を可視化 → 過負荷の防止
```

日付がなければ逆算できない。逆算できなければ、行動計画は「なんとなく動く」に戻る。

### 「とりあえず半年後」問題への対処

日付を必須にすると、担当者が意味のない遠い日付を入れることがある。これを防ぐのが **push count（期ずれ回数）** の仕組みだ。

想定受注日を後ろ倒しにするたびに活動ログに `close_date_change` が自動記録され、回数がカウントされる。

| push count | 意味 |
|-----------|------|
| 0〜1回 | 初回見立て通り、または軽微な修正 |
| 2回 | 要注意。フェーズ進捗と照合を推奨 |
| 3回以上 | クロージング精度または初期見極めに課題の可能性 |

「とりあえず半年後」を入れた担当者は、そのまま半年後に近づいても進捗がなければ日付を延ばし続ける。その延ばした事実が数字として残る。

### 管理者は「見立て精度」を数字で管理する

管理者がヨミ会でやるべきことは「なぜ受注できなかったか」の事後検討ではなく、**「この見立ては今のPhaseとBANTと整合しているか」の事前チェック**だ。

- Phase 1（見積書未送付）なのに来月受注と書いてある → 根拠を確認
- push count が 3 の案件が複数ある担当者 → 初期見極めの訓練が必要
- 想定受注日を入れることを嫌がる担当者 → 「コミットしたくない」心理の現れ

このSFAの指導ダッシュボード（`#coach`）では、push count をチームおよび担当者別に集計し、「見立て精度」という定量スキル指標として可視化する。数値があれば、マネージャーは感情論ではなく事実から話を始められる。

### 日付がないと壊れる機能

想定受注日を任意にした瞬間に、以下の機能が動作しなくなる。

| 機能 | 理由 |
|------|------|
| 今すぐやること（マイページ） | closeDate から残日数を計算できない |
| ヨミ会（forecast） | 月次・四半期の着地予測が出せない |
| 期ずれトラッキング（push count） | 比較元の日付がないと変化を検出できない |
| 停滞アラート（coach） | Phase が動いていない期間の重みが測れない |

「Phase 0 の探索段階では日程が見えない」は正しい。だがその場合でも、「3ヶ月後に判断する」という日付を入れることはできる。確定日でなくてよい。**現時点の見立てを明示することが、管理の起点になる。**

## 設計判断：フィールドレベルACLを実装しない理由

「ロールや部署によって、編集できるフィールド・参照のみのフィールド・非表示のフィールドを分けたい」という要求は、SFAでは必ず上がる。しかし、このシステムではあえて実装していない。

### フィールドACLが引き起こすコスト

フィールドACLを実装すると、「ロール × フィールド → 権限」のマトリクスを **3つの層で同期し続ける義務** が生まれる。

```
画面（フォームのdisabled/hidden）
  ↕ ずれると表示上は制御できても保存できてしまう
API（バリデーション）
  ↕ ずれると直接APIを叩かれると突破される
エクスポート・レポート（CSV出力、ダッシュボード集計）
  ↕ ずれると「見えないはずの数字」が別経路で見える
```

この同期は機能追加のたびに必要になる。「CSVエクスポートを足す」「AIに要約させる」「Slackに通知する」——いずれも権限チェックを追加しなければ穴になる。メンテする人間が変わったとき（あるいはAIが改造するとき）、マトリクスの存在に気づかず漏れを作る可能性が高い。

### 本当に防ぎたいことを分解する

「フィールドACLが欲しい」という声の裏にある要求を分解すると、たいてい以下のどれかに帰着する。

| 本当に防ぎたいこと | このSFAでの対応 |
|-------------------|----------------|
| 受注後に誰かが金額を書き換える | **ロック条件**で案件ごと凍結。`isLocked()` 1関数で全経路を制御 |
| salesが他人の案件を編集する | 画面レベルのロールアクセス制御（実装済み） |
| 部長に担当者の個別金額を見せたくない | 集計ビューを別途作る。フィールド隠しより意図が明確 |
| 経営幹部に進捗フェーズを見せたくない | ダッシュボードは集計値のみ表示する設計で対応済み |

「フィールドACLで解決できる問題」は、より単純な別の手段で解決できる場合がほとんどだ。

### ロック条件による代替

このSFAでは「案件ごとの凍結」を採用している。

```js
// constants.js — ここ1箇所を見れば権限の全体像がわかる
export function isLocked(deal, cfg) {
  return LOCK_TRIGGER_DEFS.some(t => t.check(deal, cfg))
}
```

admin がトリガーを設定し（受注済み・失注済み・Phase到達・期日超過）、条件を満たした案件は全フィールドが参照のみになる。将来APIやエクスポートを追加するときも、`isLocked()` を呼ぶだけで権限チェックが完結する。

### フィールドACLを実装すべき条件

以下を**すべて満たす**場合は、フィールドACLの導入を検討する価値がある。

1. フィールド単位の制御が法令・コンプライアンス要件として明文化されている
2. サーバー側APIが存在し、API層でも同じチェックを強制できる
3. 権限マトリクスを継続的にメンテするオーナーが決まっている

イントラネット・デモ用途のこのSFAでは、上記をいずれも満たさない。

## よくある質問（運用 FAQ）

### Q. スタッフの所属部署が変わると何が起きますか？

**影響があるもの・ないもの**

| データ | 影響 | 理由 |
|--------|------|------|
| 過去の案件の部署・担当者表示 | **なし** ✅ | 案件レコードに `dept_name` / `assignee_name` を作成時点でスナップショット保存しているため |
| ダッシュボードの受注集計 | **なし** ✅ | `deal.dept_id`（案件作成時点の所属）で集計するため |
| 目標設定（四半期目標） | **なし** ✅ | 担当者の `user_id` をキーに保存しているため |
| 指導ダッシュボードの部署フィルター | **影響あり** ⚠️ | ユーザーマスターの現在の `dept_id` で担当者を絞り込むため、移動後は新部署にしか表示されない |
| 担当者別目標配分画面 | **影響あり** ⚠️ | 同上。旧部署の配分画面からは姿が消える |

**影響のある画面のイメージ**

```
移動前：田中が「東日本営業部」フィルターに表示されていた
移動後：田中が「西日本営業部」フィルターに移動
        → 東日本フィルターに田中の案件（deal.dept_id は旧部署のまま）は残るが
          指導ダッシュボードの担当者一覧には田中が出なくなる
```

**対応方法**

1. **部署移動の前に**、旧部署での四半期実績を指導ダッシュボードでスクリーンショット or メモしておく
2. マスター管理 → ユーザー管理 で当該スタッフの所属部署を新部署に変更する
3. 移動後は新部署のフィルターで管理を継続する

> 過去の受注金額・案件数・活動ログはすべて案件レコード側に保持されるため、消えることはありません。指導ダッシュボードの「どの部署フィルターで見えるか」が変わるだけです。

---

### Q. 部署を廃止・統合した場合はどうなりますか？

マスター管理で部署を「無効化」すると新規案件の選択肢から消えますが、既存の案件レコードには `dept_id` が残ります。ダッシュボードの部署フィルターからも廃止部署は消えるため、その部署の過去案件は「全部署」表示でのみ集計されます。

**対応方法**

- 廃止前に当該部署の実績をダッシュボードで確認・記録しておく
- 統合先の部署に引き継ぐ案件は、案件編集画面で `部署` を変更して保存する（保存時点から新部署扱いになる）

## AI による改造ガイド

このコードはAIが改造しやすいよう設計されています。

- **ドメインルールを変えたい** → `constants.js` の `PHASES` / `BANT_ITEMS` / `YOMI_RULES` を編集
- **画面を追加したい** → `views/` に新ファイルを作成し、`app.js` のルーターに追加
- **バックエンドDBに移行したい** → `data.js` の関数シグネチャを維持しつつ、実装をfetch APIに差し替え
- **認証を強化したい** → `login.js` のログイン処理とサーバー側エンドポイントを追加

## ⚠️ セキュリティについて

このSFAは社内イントラネット・デモ用途を想定しています。
認証はlocalStorage＋JSロール判定のため、URLの直接アクセス制御はできません。
本番運用時はサーバー側認証（セッション / JWT）とDBを追加してください。

パスワードはブラウザのSubtleCrypto APIでSHA-256ハッシュ化していますが、
ソルトなしのハッシュはレインボーテーブル攻撃に対して脆弱です。
本番環境ではbcrypt等のKey Derivation Functionをサーバー側で使用してください。
