# 改造 TODO リスト

## マルチテナント化

### 概要

現在はシングルテナント（1つのSQLite DBに全ユーザー・全データが混在）。
複数の事業者が独立したデータ空間で利用できるSaaS構成に変更する。

### やりたいフロー

1. LP上の登録フォームで「グループ名」と「Gmailアドレス」を入力してPOST送信
2. バックエンドで事業者ID（`tenant_id`）を採番し、adminユーザーとして自動登録
3. 「早速使ってみてね」のメッセージとともにGoogleログイン画面へ遷移
4. ユーザーはGoogleログインで入り、必要に応じてメンバーを追加していく

### 主な実装箇所

| 変更箇所 | 内容 |
|----------|------|
| DBスキーマ | 全テーブル（users / deals / depts / activities / targets / sessions）に `tenant_id` カラム追加 |
| 全クエリ | WHERE句に `tenant_id = ?` を追加してテナント間データ分離 |
| セッション | ログイン後のセッションにテナントコンテキストを保持 |
| 登録API | `POST /register`：グループ名・Gmail受付、テナント作成、adminユーザー登録 |
| 認証フロー | Googleログイン後にテナントを特定する仕組み（1メール = 1テナントの場合は単純、複数テナント参加は別設計が必要） |
| LP | 登録フォームの追加 |

### 設計上の検討事項

- 1つのGmailアドレスが複数テナントに所属できるか？
- テナントデータをSQLiteで分離するか、テナントごとに別ファイルにするか
- 無料プラン上限（ユーザー数・案件数）を設けるか
